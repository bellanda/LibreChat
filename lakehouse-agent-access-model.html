Modelo de acesso: Lakehouse + Agentes de IA

Arquitetura de permissões em duas camadas — dados no Lakehouse, invocação na aplicação

Fluxo 1 — Usuário invoca agente, agente consome o Lakehouse

Este é o fluxo principal e cobre a maioria dos casos de uso: analistas, engenheiros, supervisores e gestores acessando agentes especializados por domínio.

Como funciona

O Lakehouse já possui um sistema completo de autenticação e autorização. Qualquer consumer — humano ou agente — precisa enviar um Bearer JWT em cada requisição. O token, por si só, autentica e entrega o UserID para o backend de maneira segura. A partir daí, o backend cruza o UserID com a lista de roles e permissions, e verifica se o recurso solicitado bate com algum pattern de acesso.

Para os agentes, a lógica é idêntica: cada agente é um User dentro do Lakehouse. O Agente Financeiro é o UserID 25, o Agente de Pós-vendas é o UserID 31, e assim por diante. Criar um novo agente é criar um novo user com as permissions corretas — zero lógica nova de autorização.

Camada 1 — Permissão de dados

Controlada pelo UserID do agente no Lakehouse. O agente só enxerga o que o user dele permite. Patterns definem o escopo: tabelas inteiras, subdivisões, análises pré-computadas, documentos.

Camada 2 — Permissão de invocação

Controlada na aplicação (API do chat/orquestrador). Quem pode chamar o Agente Financeiro é uma ACL simples: user → [agentes]. Desacoplada dos dados.

O que o agente tem acesso (exemplos)

Quando bem construído, cada agente tem à disposição um conjunto rico de dados em tempo real e alto poder de análise:

bronze/vehicles/* → dados brutos ingestados silver/vehicles/* → dados limpos e tipados gold/vehicles/* → tabelas completas prontas para análise diamond/vehicles/* → se necessário, camada adicional gold/vehicles/kpi_daily → estatísticas pré-calculadas do dia gold/vehicles/kpi_weekly → estatísticas pré-calculadas da semana gold/vehicles/kpi_monthly → estatísticas pré-calculadas do mês diamond/vehicles/resumo_metricas.json → JSON de métricas para o agente coletar atualizado (dia, semana, mês, ano) gold/vehicles/tabela5 → tabela completa, agente conhece colunas e lógica de negócio via metadados /api/v1/lakehouse/exports/sql → executa SQL via DuckDB em sandbox /api/v1/lakehouse/exports/polars → executa Polars code em sandbox /api/v1/lakehouse/files/* → download de docs não estruturados

Os endpoints aceitam o campo client: "agent" ou client: "human", permitindo diferenciar a origem da requisição para logging e rate limiting.

Pattern matching de permissões

O sistema de permissões já suporta granularidade variável via glob patterns:

gold/* → acesso total à camada gold gold/vehicles/* → acesso a tudo dentro do domínio vehicles gold/vehicles/tabela_x → acesso somente a 1 tabela específica gold/*_kpi_daily* → acesso a qualquer KPI diário, cross-domain diamond/finance/sub/* → acesso a subdivisão específica

Princípio: o agente nunca sai do perímetro do seu próprio UserID. Se o agente não tem pattern para um recurso, recebe 403 — exatamente como um humano receberia.

Fluxo 2 — Orquestrador para perfis executivos

O orquestrador não é uma nova arquitetura — é o mesmo Fluxo 1 composto. Ele existe para o caso específico em que um VP, CEO, Conselho ou diretores precisam de informações de múltiplos domínios simultaneamente, sem acessar cada agente individualmente.

Quando usar o orquestrador

O orquestrador resolve um caso específico: um perfil executivo precisa de visão cross-domain. Um CEO pergunta "como estão as vendas, o financeiro e o marketing este mês?" — essa pergunta cruza 3 domínios. Em vez de dar acesso direto a 3 agentes e deixar o humano coordenar, o orquestrador recebe a pergunta, decompõe, delega para os agentes filhos, e consolida a resposta.

Cada agente filho executa o Fluxo 1 completo com seu próprio JWT — o orquestrador não bypassa nenhuma permissão. Ele só compõe resultados.

Cenários: CEO, VP, Conselho → acesso ao Orquestrador Executivo (agrega Financeiro + Pós-vendas + Marketing) Diretor Comercial → acesso ao Orquestrador Comercial (agrega Pós-vendas + Marketing) Analista de pós-vendas → acesso direto ao Agente Pós-vendas Engenheiro de dados → acesso direto ao Agente Financeiro Supervisor de marketing → acesso direto ao Agente Marketing

A cascata funciona naturalmente: o executivo com acesso ao orquestrador herda o alcance combinado dos agentes filhos, sem configuração adicional por domínio.

Ponto de atenção — rastreabilidade: quando o orquestrador chama um agente filho, o log precisa registrar quem originou a cadeia. Propagar um campo on_behalf_of ou trace_id em cada chamada resolve. Sem isso, o audit trail mostra apenas "UserID 25 acessou gold/finance/*" — mas não mostra que foi o CEO via orquestrador que disparou a consulta.

Resumo da arquitetura

Lakehouse (já pronto)

bronze silver gold diamond

JWT auth, roles, permissions, pattern matching por tabela/análise/arquivo. Endpoints SQL DuckDB, Polars sandbox, file download. Campo client: agent|human.

Agentes (= Users no Lakehouse)

Cada agente tem UserID próprio com permissions específicas. Acessa dados pré-computados (KPIs diários, semanais, mensais), tabelas completas com metadados de colunas, queries SQL/Polars, documentos e contexto de negócio.

Camada 2 — Invocação

ACL na aplicação define quais agentes cada usuário humano pode chamar. Completamente desacoplada dos dados.

Orquestrador (opcional)

Compõe múltiplos agentes para perfis executivos. Não cria permissões novas — delega para agentes filhos que executam o fluxo padrão com seus próprios JWTs.
